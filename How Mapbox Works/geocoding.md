<p>The <a href="https://docs.mapbox.com/api/search/#geocoding" rel="nofollow">Mapbox Geocoding API</a> performs two main tasks: <em>forward geocoding</em> and <em>reverse geocoding</em>. Forward geocoding converts text into geographic coordinates, for example, turning <code>2 Lincoln Memorial Circle NW</code> into <code>-77.050,38.889</code>. Reverse geocoding converts geographic coordinates into a text description, for example, turning <code>-77.050,38.889</code> into <code>2 Lincoln Memorial Circle NW</code>.</p>
<p>All geocoding requests require you to submit a <em>query</em>, or what you're trying to find. When you make a query, you get a <em>response</em>, a JSON-formatted document of the most relevant results from your query. This guide provides an overview of how the Geocoding API works, how to use it, how to provide feedback, and links to relevant documentation to get you started.</p>
<p>{{

}}</p>
<p>Here's an example that uses the <a href="https://github.com/mapbox/mapbox-gl-geocoder">Mapbox GL geocoder plugin</a> to add forward geocoding to a web app built with Mapbox GL JS. Type in a location to view the raw JSON response for that query. Geocoding results are returned in <a href="https://github.com/mapbox/carmen/blob/master/carmen-geojson.md">Carmen GeoJSON</a> format and contain both descriptive location information and geographic information. You can use the geographic information returned in the JSON response to show your query on a map, like in this example, or do additional spatial analysis.</p>
<h2><a id="user-content-how-geocoding-works" class="anchor" aria-hidden="true" href="#how-geocoding-works"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>How geocoding works</h2>
<p>The <a href="https://docs.mapbox.com/api/search/#geocoding" rel="nofollow">Mapbox Geocoding API</a> has two distinct parts: the <strong>source data</strong> we use to define locations and the <strong>tools</strong> we use to search for and return those locations.</p>
<h3><a id="user-content-source-data" class="anchor" aria-hidden="true" href="#source-data"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Source data</h3>
<p>The Mapbox Geocoding API contains <a href="https://www.mapbox.com/about/maps/#data-sources" rel="nofollow">data sources</a> from governments, open data projects, and private companies. In some cases, results from the Geocoding API may differ from <a href="https://www.mapbox.com/vector-tiles/mapbox-streets-v7/" rel="nofollow">Mapbox Streets</a> or OpenStreetMap data.</p>
<p>The Mapbox Geocoding API source data contains the following types of geographic information, ordered from the most granular to the largest:</p>
<ul>
<li><strong>Points of interest (POI):</strong> A named place including commercial businesses, public buildings, monuments, and parks, among other features.</li>
<li><strong>Address:</strong> A specific mailing address, including the address number if applicable.</li>
<li><strong>Neighborhood:</strong> A colloquial name for a smaller area within a place. Neighborhoods do not necessarily have specific, legally defined boundaries. Only present in some countries.</li>
<li><strong>Locality:</strong> An administrative unit that is smaller than a place. Only present in some countries.</li>
<li><strong>Postcode:</strong> A geographic area of the address component used for sorting mail.</li>
<li><strong>Place:</strong> Cities, towns, and villages. Note that some large cities (such as Tokyo and Istanbul) may be categorized as regions rather than places.</li>
<li><strong>District:</strong> An administrative unit that is larger than a place but smaller than a region. Only present in some countries.</li>
<li><strong>Region:</strong> States, provinces, and prefectures. This is typically the largest sub-national administrative unit of a country. Note that some large cities (such as Tokyo and Istanbul) may be categorized as regions rather than places.</li>
<li><strong>Country:</strong> Generally recognized countries or, in some cases like Hong Kong, an area of quasi-national administrative status that has been given a designated country code under ISO 3166-1.</li>
</ul>
<p>This hierarchy of feature types is also used to determine what will be returned as the encompassing parent features in a Geocoding API response object's <a href="https://docs.mapbox.com/api/search/#geocoding-response-object" rel="nofollow"><code>context</code> property</a>. For example, if the returned feature is a <code>place</code> (like Detroit), then the encompassing parent features in the <code>context</code> property will be the <code>region</code> (the state of Michigan) and the <code>country</code> (United States).</p>
<h3><a id="user-content-tools" class="anchor" aria-hidden="true" href="#tools"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Tools</h3>
<p>The Mapbox Geocoding API uses <a href="https://github.com/mapbox/carmen">Carmen</a>, an open source project for Mapnik vector tile-based geocoding. For more on Carmen, see <a href="https://github.com/mapbox/carmen#how-does-carmen-work">How does Carmen work?</a>.</p>
<h3><a id="user-content-customizing-your-queries" class="anchor" aria-hidden="true" href="#customizing-your-queries"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Customizing your queries</h3>
<p>The Mapbox Geocoding API accepts many optional parameters. You can use these optional parameters to customize your queries so that the most relevant results are returned. These parameters can be specified using URL query parameters, or they can be specified as options with one of our client-side libraries or plugins. These parameters allow you to view results as you type, filter results by geographic feature type, and limit or bias results to a specified area. For example, to limit your search results to addresses in the Washington DC Metro area, you could set the <code>type</code> parameter to <code>address</code> and the <code>bbox</code> parameter to <code>-77.08,38.90,-76.99,38.95</code>. With those parameters set, your query for <code>Constitution Ave</code> will only return street addresses in the DC Metro area, and will not include features you're not interested in (like <code>Constitution Ave, El Paso, Texas 79908, United States</code>).</p>
<p>Read the <a href="https://docs.mapbox.com/api/search/#geocoding" rel="nofollow">Geocoding API documentation</a> for more information on available features.</p>
<h3><a id="user-content-language-support" class="anchor" aria-hidden="true" href="#language-support"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Language support</h3>
<p>The Mapbox Geocoding API accepts a <code>language</code> query parameter, which allows you to specify the language in which you would like to search. One or more languages can be specified using <a href="https://en.wikipedia.org/wiki/List_of_ISO_639-1_codes" rel="nofollow">ISO 639-1 codes</a>. Translation availability varies by language and region, and you can expect more consistent results for areas where the specified language is most widely used. Language support has three different levels:</p>
<ul>
<li><strong>Global coverage.</strong> These languages are almost always present for <code>country</code>, <code>region</code>, and prominent <code>place</code> features.</li>
<li><strong>Local coverage.</strong> These languages may lack global coverage, but they are almost always present for <code>country</code>, <code>region</code>, and prominent <code>place</code> features where they are widely used.</li>
<li><strong>Limited coverage.</strong> These languages are sometimes present, but coverage tends to be inconsistent or geographically constrained.</li>
</ul>
<p>For a current list of the languages covered at each level, see the <a href="https://docs.mapbox.com/api/search/#language-coverage" rel="nofollow">Geocoding API documentation</a>.</p>
<h3><a id="user-content-search-result-prioritization" class="anchor" aria-hidden="true" href="#search-result-prioritization"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Search result prioritization</h3>
<p>Results from geocoding queries are prioritized differently depending on whether the request was a <em>forward geocode</em> or a <em>reverse geocode</em>.</p>
<h4><a id="user-content-result-prioritization-in-forward-geocoding" class="anchor" aria-hidden="true" href="#result-prioritization-in-forward-geocoding"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Result prioritization in forward geocoding</h4>
<p>When a forward geocoding query (human-readable text like âSan Franciscoâ) is submitted to the Mapbox Geocoding API, the geocoder uses Carmen, a text search engine, to search our spatial indexes for features that match the userâs query. (For a more detailed description of how the Carmen text search engine works, read the <a href="https://github.com/mapbox/carmen/blob/master/docs/how-carmen-works.md">Carmen documentation</a>.) The geocoder applies filters on the backend to sort the results that match, or partially match, the searched text. These filters are <em>textual relevance</em> and <em>prominence score</em>.</p>
<h5><a id="user-content-textual-relevance" class="anchor" aria-hidden="true" href="#textual-relevance"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Textual relevance</h5>
<p>When the geocoder sorts and prioritizes results, the first filter that is applied is <code>relevance</code>. <code>relevance</code> is a value that indicates how well a feature in our dataset matches the query. This property is surfaced in the Geocoding API response, and is a numerical score from <code>0</code> (the result does not match the queried text at all) to <code>1</code> (the result matches the queried text most completely). You can use the <code>relevance</code> property to remove results that don't fully match the query.</p>
<p>In this example query for the address â515 15th St NW, Washington, DC 20004â, the expected result is first in the response with a <code>relevance</code> of <code>0.875</code>. Other results in other cities, since they donât match the search text as closely, have a <code>relevance</code> score of <code>0.2</code>.</p>
<pre><code>https://api.mapbox.com/geocoding/v5/mapbox.places/515%2015th%20St%20NW%2C%20Washington%2C%20DC%2020004.json?types=address&amp;access_token={{ &lt;UserAccessToken /&gt; }}
</code></pre>
<h5><a id="user-content-prominence-score" class="anchor" aria-hidden="true" href="#prominence-score"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Prominence score</h5>
<p>In the case that multiple features have the same <code>relevance</code> score, a second filter called <code>score</code> is applied. This is based on the popularity or prominence of a feature. For example, a search for âParisâ will equally match âParis, Franceâ and âParis, Texasâ â theyâll have the same <code>relevance</code> score. The <code>score</code> filter helps break this tie on the backend, and surfaces âParis, Franceâ first since it is a more popular feature:</p>
<pre><code>https://api.mapbox.com/geocoding/v5/mapbox.places/paris.json?access_token={{ &lt;UserAccessToken /&gt; }}
</code></pre>
<h4><a id="user-content-result-prioritization-in-reverse-geocoding" class="anchor" aria-hidden="true" href="#result-prioritization-in-reverse-geocoding"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Result prioritization in reverse geocoding</h4>
<p>For reverse geocodes, results at a given set of coordinates are sorted by order of <strong>spatial hierarchy</strong>. For example, a more granular feature such as an <code>address</code>, <code>poi</code>, or <code>postcode</code> will return first in the response before feature types like <code>region</code>  or <code>country</code>. The full spatial hierarchy, ordered from the most granular to the largest, is: <em>point of interest (POI)</em>, <em>address</em>, <em>neighborhood</em>, <em>locality</em>, <em>postcode</em>, <em>place</em>, <em>district</em>, <em>region</em>, and <em>country</em>. (For more details about these geographic information types, see the <a href="#source-data">Source data section</a>.)</p>
<p>This reverse geocoding query example returns features closest to the query point (in the case of point features like <code>address</code> and <code>poi</code>) and features that contain the query point (in the case of polygon features like <code>place</code> or <code>region</code>), in order of hierarchy from the most granular (address or POI) to the largest feature (country):</p>

</code></pre>
